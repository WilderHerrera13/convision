<threat_model>
## Threat Model

**Threats considered:**
- `laboratory.Service.UploadEvidence` takes `io.Reader` and `filestore.Storage` — forgetting to mock `MockStorage.Upload` causes a nil pointer panic — **Mitigation:** Always set up `mockStorage.On("Upload", ...)` with a plausible return value before calling `UploadEvidence`; use `MockStorage` from `testutil/mocks`.
- `catalog.Service` takes 7 repo arguments — constructing it in tests with `nil` repos for unused entities causes panics if the service indexes them without nil checks — **Mitigation:** Pass `MockBrandRepository` (etc.) for all 7 repos even in tests that only exercise one entity type; all un-exercised mock repos will receive no `On()` calls and will not be called.
- `cashclose.Service` state machine uses `domain.CashRegisterCloseStatus*` constants — using string literals in test assertions instead of constants can make tests pass today but fail after a rename — **Mitigation:** Always assert using `domain.CashRegisterCloseStatusDraft`, `domain.CashRegisterCloseStatusSubmitted`, `domain.CashRegisterCloseStatusApproved` constants.
- `inventory.Service.CreateTransfer` validation checks source != destination at the service level — if Phase 8 moved this to the repository, tests may need to mock the repo to return an error — **Mitigation:** Read the current `inventory/service.go` before writing tests; adapt to the actual validation site.

**Risk level:** Low (test code only, no production access)
</threat_model>
