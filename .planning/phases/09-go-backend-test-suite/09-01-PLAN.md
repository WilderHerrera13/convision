# Plan 09-01: Test Infrastructure

## Objective

Create the `internal/testutil/` package with all shared test utilities, hand-written mock implementations for every domain Repository interface, and HTTP test helpers needed by plans 09-02 through 09-05. This plan is wave 1 and has no dependencies; all subsequent plans depend on it.

## must_haves
- [ ] `github.com/stretchr/testify v1.11.1` appears as a direct dependency in `go.mod`
- [ ] `go build ./convision-api-golang/internal/testutil/...` exits 0 with no errors

<threat_model>
## Threat Model

**Threats considered:**
- Test files could accidentally expose real credentials in fixtures — **Mitigation:** Use fake IDs and placeholder emails (`test@example.com`) in all test data; never read from `.env` or production env vars in test helpers.
- Mock implementations could silently diverge from real interfaces — **Mitigation:** Every mock struct includes a compile-time interface check (`var _ domain.PatientRepository = (*MockPatientRepository)(nil)`) so the compiler enforces exact method signatures.
- Parallel tests sharing `os.Setenv` for `JWT_SECRET` — **Mitigation:** `JWT_SECRET` is set in each package's `TestMain`; testutil helpers do not call `os.Setenv` themselves.
- `testutil` package accidentally importing `platform/storage/postgres` (violates layering) — **Mitigation:** `testutil` imports only `internal/domain`, `internal/platform/auth`, `internal/platform/filestore`, and stdlib; import cycle is caught at `go build` time.

**Risk level:** Low (test-only code, no production data access, no DB connection)
</threat_model>
