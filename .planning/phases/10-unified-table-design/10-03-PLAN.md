# Plan 10-03: Admin entity lists batch 2 — Patients (admin), Quotes, Sales, Purchases

**Phase:** 10 — Unified Table Design — Entity Tables
**Plan:** 03 of 05
**Wave:** 2
**Depends on:** Plan 10-01 (EntityTable defaults must be updated first)
**Estimated scope:** 4 pages to migrate (high complexity — large files with embedded dialogs)

---

## Goal

Migrate four complex admin entity list pages to use the centralized `EntityTable` component. These pages are large (1000–2500 lines) because they contain multiple embedded dialogs. The migration targets ONLY the table + pagination + query block; dialogs and their state remain untouched.

## Success Criteria

- [ ] `Patients.tsx` (admin) uses `<EntityTable>` for its patient list
- [ ] `Quotes.tsx` (admin) uses `<EntityTable>` for its quote list
- [ ] `Sales.tsx` (admin) uses `<EntityTable>` for its sale list
- [ ] `Purchases.tsx` uses `<EntityTable>` for its purchase list
- [ ] Zero new TypeScript errors introduced by the migration in any of the four files
- [ ] `grep -n "useState.*page\b" convision-front/src/pages/admin/Patients.tsx` returns 0 table-state matches
- [ ] Existing filter UI (status dropdowns, date range pickers) above the table continue to work — passed as `extraFilters`

<task type="sequential">
  <name>Migrate Patients.tsx (admin) to EntityTable</name>
  <read_first>
    - convision-front/src/pages/admin/Patients.tsx (full file — canonical reference, READ COMPLETELY)
    - convision-front/src/components/ui/data-table/EntityTable.tsx (props — especially extraFilters and toolbarLeading/Trailing)
    - convision-front/src/services/patientService.ts (or equivalent service for patient list)
  </read_first>
  <action>
    IMPORTANT: The admin Patients page is the CANONICAL reference. Read it carefully before changing it.

    The patients page currently:
    - Uses DataTable with manually managed page/search/filter state
    - Has a custom toolbar above the table (search + status filter + CTA button)
    - Has custom pagination below the table
    - Contains multiple dialogs (create, edit, delete patient)

    Migration steps:
    1. Read the entire file to understand the current structure
    2. Identify all table-specific state variables (`page`, `perPage`, `search`, `filterStatus`) and the `useQuery` call for the patient list
    3. Remove those state variables and the table `useQuery`
    4. Keep the dialogs and their state untouched
    5. Replace the DataTable + custom pagination + toolbar block with:

    ```tsx
    <EntityTable<Patient>
      columns={columns}
      fetcher={(params) => patientService.list({
        page: params.page,
        per_page: params.per_page,
        search: params.search,
        ...(filterStatus !== 'all' ? { status: filterStatus } : {}),
      })}
      queryKeyBase="patients-admin"
      extraFilters={{ status: filterStatus !== 'all' ? filterStatus : undefined }}
      searchPlaceholder="Buscar paciente..."
      toolbarLeading={
        <div className="flex flex-col gap-0.5">
          <span className="text-[14px] font-semibold text-[#121215]">Pacientes</span>
          <span className="text-[11px] text-[#7d7d87]">Base de datos de pacientes</span>
        </div>
      }
      toolbarTrailing={
        <>
          {/* Status filter select if compact, or render above EntityTable */}
          <Button
            onClick={() => setIsCreateOpen(true)}
            className="h-[34px] rounded-[6px] bg-[#8753ef] px-4 text-[12px] font-semibold text-white hover:bg-[#7040d8]"
          >
            + Nuevo paciente
          </Button>
        </>
      }
      emptyStateNode={<EmptyState variant="patients" onAction={() => setIsCreateOpen(true)} />}
      filterEmptyStateNode={<EmptyState variant="table-filter" />}
    />
    ```

    Note: `filterStatus` state stays at page level for the filter dropdown UI. It is passed to `extraFilters` so EntityTable re-fetches when it changes.

    If the current page renders the status filter as a row ABOVE the table card, keep that row in place and connect it to `extraFilters`. Do NOT move it into the toolbar unless it fits compactly.
  </action>
  <acceptance_criteria>
    - `grep -n "EntityTable" convision-front/src/pages/admin/Patients.tsx` returns at least 1 match
    - `grep -n "useState.*page\b\|useState.*perPage" convision-front/src/pages/admin/Patients.tsx` returns 0 table-state matches
    - `grep -n "useQuery.*patient\|useQuery.*Patient" convision-front/src/pages/admin/Patients.tsx` returns 0 list-query matches (mutation queries for dialogs are OK)
    - `npx tsc --noEmit 2>&1 | grep "Patients.tsx" | grep "admin"` returns no errors
  </acceptance_criteria>
</task>
