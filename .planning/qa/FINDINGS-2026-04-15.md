### Resolución (2026-04-15)

### QA-003
- Rol: receptionist
- URL: http://localhost:4300/receptionist/appointments (lista) y http://localhost:4300/receptionist/appointments/1 (detalle)
- Severidad: **mayor** (dato incorrecto visible al usuario)
- Pasos:
  1. Crear una cita con hora 10:00 (via API o modal)
  2. Ver la lista de citas del día
  3. Abrir el detalle de la cita
- Esperado: La hora mostrada es 10:00 a.m. (hora local)
- Observado: La hora muestra **05:00 a. m.** tanto en lista como en detalle — 5 horas menos que la hora ingresada
- Evidencia: API devuelve `"scheduled_at": "2026-04-15T10:00:00.000000Z"`. Pantalla de lista muestra `05:00`. Detalle muestra `Hora: 05:00 a. m.`
- Causa raíz: El frontend renderiza el timestamp UTC directamente sin convertir a zona horaria local (Colombia = UTC-5). El timestamp `10:00 UTC` se interpreta como `05:00` local
- Estado: **confirmado**

### Resolución (2026-04-15)
- Función añadida: `parseLocalDatetime()` en `convision-front/src/lib/utils.ts` — parsea el ISO string del backend quitando la `Z`/offset para evitar la conversión UTC→local
- Archivos corregidos: `Appointments.tsx`, `AppointmentDetail.tsx`, `ReceptionistDashboard.tsx`, `SpecialistDashboard.tsx`, `SpecialistAppointmentDetail.tsx`, `AppointmentsSection.tsx`, `ClinicalEvolutionsList.tsx`
- Estado: **resuelto**

### QA-006
- Rol: receptionist
- URL: http://localhost:4300/receptionist/appointments (modal Nueva cita — paso 3)
- Severidad: **mayor** (lógica de negocio — permite agendar en el pasado)
- Pasos:
  1. Abrir modal "Nueva cita"
  2. En paso 3 (Fecha y hora), seleccionar la fecha de HOY
  3. Seleccionar un horario de la mañana (ej. 10:00) cuando son las 17:35
  4. Confirmar la cita en el paso 4
- Esperado: El sistema rechaza o advierte que el horario ya pasó (10:00 AM cuando son 17:35)
- Observado: La cita se crea exitosamente con fecha/hora en el pasado. El resumen muestra "miércoles, 15 de abril de 2026 - 10:00" sin ninguna advertencia. API confirma creación con status `scheduled`
- Evidencia: Cita ID 2 creada vía API con `scheduled_at: "2026-04-15T10:00:00.000000Z"` cuando la hora de creación fue `17:35:14Z`
- Estado: **confirmado**

### Resolución (2026-04-15)
- Frontend (`convision-front/src/components/appointments/NewAppointmentDialog.tsx`):
  - Añadida función `isTimeInPast(date, time)` que compara la hora seleccionada vs. `now` cuando la fecha seleccionada es hoy
  - El botón "Siguiente" en paso 2 se deshabilita si `isTimeInPast` es `true`
  - Los slots de hora ya pasados aparecen tachados y deshabilitados (estilo `line-through + cursor-not-allowed`)
  - Se muestra mensaje de error en rojo cuando el usuario selecciona una hora pasada del día de hoy
  - Al cambiar de fecha, si el `selectedTime` ya pasó para el nuevo día, se limpia automáticamente
- Backend (`convision-api/app/Http/Requests/Api/V1/Appointment/StoreAppointmentRequest.php`):
  - Añadida regla `after:now` en `scheduled_at` para rechazar fechas en el pasado con HTTP 422
  - Mensaje de error en español: "La fecha y hora de la cita debe ser en el futuro."
  - Verificado via curl: API retorna 422 con mensaje correcto para fechas pasadas

### QA-008
- Rol: specialist
- URL: http://localhost:4300/specialist/appointments/2 (después de "Tomar Cita")
- Severidad: **menor** (UX — falta feedback de transición de estado)
- Pasos:
  1. Abrir detalle de cita como specialist
  2. Hacer clic en "Tomar Cita"
- Esperado: Toast/notificación de éxito indicando que la cita cambió a estado "En progreso". Los nuevos botones ("Crear Evolución Primero", "Crear Fórmula") se hacen visibles con scroll automático hacia ellos
- Observado: El botón "Tomar Cita" desaparece (cambia a "Tomando..." brevemente), los nuevos botones aparecen pero NO hay toast de confirmación. El estado de la cita cambia en API a `in_progress` pero el badge de estado en la UI no es claramente visible (aparece como un pequeño indicator amarillo parcialmente cortado por el layout)
- Evidencia: Snapshot post-acción muestra estado `in_progress` en API; UI sin toast de confirmación

### Resolución (2026-04-15)
- El toast ya estaba implementado en `handleTakeAppointment`. La causa raíz era que `fetchAppointment()` llamaba `setLoading(true)` causando un re-render que destruía el toast antes de que fuera visible.
- Archivo: `convision-front/src/pages/specialist/SpecialistAppointmentDetail.tsx`
- Cambios: Añadido estado `refreshing` separado; `fetchAppointment(silent=false)` — cuando `silent=true` usa `setRefreshing` en lugar de `setLoading`, evitando el re-render destructivo del componente. Todas las llamadas post-acción ahora usan `fetchAppointment(true)`.
- Estado: **resuelto**

### QA-009
- Rol: specialist
- URL: http://localhost:4300/specialist/patients/1/history
- Severidad: **mayor** (dato incorrecto — identificación del paciente no visible)
- Pasos:
  1. Desde detalle de cita, hacer clic en "Historia" o "Crear Evolución Primero" → "Crear Historia Clínica"
  2. Observar la sección del paciente en la Historia Clínica
- Esperado: Se muestra la identificación del paciente (CC 123456789) de forma consistente con lo que muestra el detalle de cita
- Observado: La Historia Clínica muestra "Sin identificación registrada" aunque la API devuelve `identification: "123456789"` para el paciente
- Evidencia: API `GET /api/v1/appointments/2` → `patient.identification = "123456789"`, `patient.identification_type = null`. El componente de Historia Clínica muestra "Sin identificación registrada" porque `identification_type` es `null`
- Causa raíz hipotética: El componente de Historia Clínica probablemente renderiza `"{identification_type} {identification}"` y cuando `identification_type` es null, muestra el mensaje de fallback en lugar de solo la identificación
- Estado: **confirmado**

### Resolución (2026-04-15)
- Archivo: `convision-front/src/components/patients/PatientInfo.tsx`
- Causa raíz confirmada: línea 73 tenía condición `patient.identification_type && patient.identification` — cuando `identification_type` es `null`, la condición falla aunque haya `identification`.
- Cambio: Condición separada en dos niveles: si hay `identification`, mostrar; si además hay `identification_type`, prefijar con el tipo. Fallback solo si no hay `identification`.
- Estado: **resuelto**
