# QA FINDINGS — Flujo Cierre de Caja
**Fecha:** 2026-04-18  
**Alcance:** Flujo completo de Cierre de Caja — backend Go API (:8001) + frontend (:4300)  
**Roles explorados:** admin, receptionist  
**Entorno:** Docker — Go API convision_go_api, Front convision-front, Postgres convision-postgres

---

## Resumen Ejecutivo

El flujo de cierre de caja está **parcialmente bloqueado**: el backend Go solo implementa los dos endpoints de lectura (`GET /list` y `GET /:id`). Faltan **8 endpoints** que el frontend ya consume (POST crear, PUT actualizar, POST submit, POST approve, POST return, PUT admin-actuals, GET advisors-pending, GET calendar). Adicionalmente, el schema de la tabla `cash_register_closes` no almacena `payment_methods` ni `denominations` como columnas, por lo que siempre se retornan vacíos.

### GOQA-CAJA-001
- **Rol:** receptionist
- **URL:** `http://localhost:4300/receptionist/cash-closes`
- **Severidad:** bloqueante
- **Pasos:**
  1. Login como `receptionist@convision.com`
  2. Navegar a `/receptionist/cash-closes`
  3. Ingresar denominaciones (3 × $100k, 2 × $50k → total $400k)
  4. Click "Guardar borrador"
  5. Click "Enviar cierre"
- **Esperado:** Borrador guardado vía `POST /api/v1/cash-register-closes` → toast "Borrador guardado"; envío vía `POST /:id/submit` → toast "Cierre enviado"
- **Observado:** Botones no producen cambio visible. Toast de error puede aparecer brevemente pero es difícil de ver. La página permanece en paso 1 (no avanza a paso 2 "Ver resumen"). El backend retorna 404 en todos los endpoints de escritura.
- **Evidencia:**
  - `curl -s -w '%{http_code}' -X POST http://localhost:8001/api/v1/cash-register-closes` → `404 page not found`
  - `curl -s -w '%{http_code}' -X POST http://localhost:8001/api/v1/cash-register-closes/1/submit` → `404 page not found`
  - `grep "cash-register" routes.go` → solo `GET ""` y `GET "/:id"` registrados
- **Estado:** confirmado

## Mapa de Endpoints Faltantes en Go API

| Endpoint | Método | Propósito | Frontend usa |
|---|---|---|---|
| `/api/v1/cash-register-closes` | POST | Crear borrador | `cashRegisterCloseService.create()` |
| `/api/v1/cash-register-closes/:id` | PUT | Actualizar borrador | `cashRegisterCloseService.update()` |
| `/api/v1/cash-register-closes/:id/submit` | POST | Enviar cierre (draft→submitted) | `cashRegisterCloseService.submit()` |
| `/api/v1/cash-register-closes/:id/approve` | POST | Aprobar cierre (admin) | `cashRegisterCloseService.approve()` |
| `/api/v1/cash-register-closes/:id/return` | POST | Devolver a borrador (admin) | `cashRegisterCloseService.returnToDraft()` |
| `/api/v1/cash-register-closes/:id/admin-actuals` | PUT | Registrar totales reales admin | `cashRegisterCloseService.putAdminActuals()` |
| `/api/v1/cash-register-closes-advisors-pending` | GET | Lista asesores con pendientes | `cashRegisterCloseService.listAdvisorsWithPending()` |
| `/api/v1/cash-register-closes-calendar` | GET | Calendario por asesor y rango | `cashRegisterCloseService.getCalendarForAdvisor()` |

## Problema de Schema DB

La tabla `cash_register_closes` NO tiene columnas para `payment_methods` ni `denominations`. El frontend los envía como arrays JSON pero el backend los ignora y siempre retorna `payment_methods: []`. Se requiere agregar columnas `JSONB` o tablas relacionadas para almacenar los medios de pago y denominaciones por cierre.

**Columnas actuales:** `id, user_id, close_date, status, total_counted, total_actual_amount, admin_actuals_recorded_at, admin_notes, advisor_notes, approved_by, approved_at, created_at, updated_at`

**Columnas requeridas (mínimo):** `payment_methods JSONB`, `denominations JSONB`
